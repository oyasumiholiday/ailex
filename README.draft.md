# Ailex — 研究プロトタイプ

> AI が生成・検証するための実験的な中間言語。
> **これは完成した言語ではなく、ひとつの仮説を検証するための研究プロトタイプです。**

---

## これは何か / これは何でないか

**何か**
- 「自己回帰生成と機械検証のために文法を最適化すると、AI にとって開発しやすくなるか？」という**仮説を検証するための最小実装**。
- 型検査器から「各生成時点で許容される次の構成子の集合」を計算し、それを LLM の constrained decoding のマスクに使う、という機構の実験台。

**何でないか**
- 実用言語・製品ではない。エディタ / IDE / パッケージ管理 / 標準ライブラリは無い（このフェーズで作る予定も無い）。
- **「世界初」を主張しない。** 関連する先行研究が多数ある（内容アドレス: Unison、typed holes: Hazel、投影編集: JetBrains MPS、契約/依存型: Idris・Agda・Dafny、多層IR: MLIR、文法制約デコード: GBNF 等）。本プロトタイプの新規性は限定的で、既存アイデアの特定の組み合わせを小さく検証するものにすぎない。
- **「Python より優れている」とは主張しない。** 測るのは優劣ではなく、「型制約が生成・修復のコストをどれだけ肩代わりできるか」。

---

## 検証しようとしている主張（断定でなく仮説）

> L1（正規形）の文法を「左から右の生成のどの接頭辞も、型が確定した部分プログラムである」ように設計すると、型検査器から各時点の許容集合を計算でき、これを constrained decoding のマスクにできる。結果として型・構文エラーが構造的に排除され、修復ループが短くなる**かもしれない**。

この主張は**反証可能**である。Ailex＋マスクが Python＋事後チェックに対して等コストで並ばなければ、（少なくとも現在の言語コアでは）主張は支持されない。

| 項目 | 状態 |
|---|---|
| L1 パーサ / 双方向型検査 / 評価器 / 修復ループ（prototype.ts） | 実装済み・動作確認済み |
| 言語コア: `if` / 比較(`>=`) / 数値リテラル（固定パレット） | 実装済み・動作確認済み |
| 型指向マスクの実証（synth.ts、訓練データ・LLM 無し） | 実装済み・動作確認済み |
| Claude Opus 駆動の逐次マスク生成（claude.ts） | 実装済み・実 LLM で動作確認済み |
| 10タスクの台（数値5＋分岐5）＋比較ハーネス（compare.ts） | 実装済み・スタンドインで配管検証済み |
| 測定機構: pass@1 / pass@k / 修復ラウンド / コスト / Python エラー内訳 / JSONログ | 実装済み |
| **実 LLM による比較の数値測定** | **実施済み（E1–E3, 2026-07-10）→ [EXPERIMENTS.md](EXPERIMENTS.md)** |

**本 README 作成時点で、実モデル（Claude Opus 等）による pass 率・コストの数値はまだ得ていない。**
ハーネスは API 鍵の無い環境ではスタンドイン（内蔵の探索ソルバ／参照解）で走り、Python 実行・採点・マスク計算・集計の**配管のみ**を検証している。数値自体は無意味である。
したがって本プロジェクトは「Claude で実証済み」とは書かない。現時点で言えるのは「配管は動く」「訓練データ・LLM 無しの素朴な列挙器なら型＋実例で穴を解ける」までである。

## 言語コアの範囲（意図的に小さい）

- 型: `Int` / `Float` / `Bool` / `Vec` / 関数。多相なし。
- 項: リテラル / 変数 / リスト / 関数適用 / `if(cond, then, els)` / ホール `?`。
- 比較は `>=`（ビルトイン）を流用。数値リテラルは**固定パレット**（Float: `-1.0, 0.0, 1.0`／Int: `0, 1`）で、任意の定数は出せない（pure-enum マスクとの意図的な妥協）。
- 現状のタスク台は 10 問（`sq`/`normsq`/`cube`/`norm`/`hypot` と、`max`/`min`/`relu`/`abs`/`sign`）。数値・ベクトルの合成と単純な分岐が中心で、依然として浅いのは既知の限界。

## 実行方法

Node 24 の型ストリップで `.ts` を直接実行（ビルド不要）。

```sh
node ailex/prototype.ts   # 修復ループの実演
node ailex/synth.ts       # 型指向マスクの実証（訓練データ・LLM 無し）
node ailex/claude.ts      # Opus 駆動の逐次マスク生成（実 API 対応・スタンドイン内蔵）
node ailex/compare.ts     # 比較実験（実 API 対応・スタンドイン内蔵）
```

実 LLM を使う場合（自分の管理下の端末で）:

```sh
cd ailex && npm install @anthropic-ai/sdk
cd .. && export ANTHROPIC_API_KEY=sk-ant-...
node ailex/compare.ts     # 駆動行が「実モデル …」になれば本番
```

`export` の代わりに `ailex/.env` に書いてもよい（`ANTHROPIC_API_KEY=sk-ant-...` の1行。`.gitignore` 済み・コミット禁止）:

```sh
echo 'ANTHROPIC_API_KEY=sk-ant-...' > ailex/.env
node ailex/compare.ts
```

`compare.ts` は環境変数で制御する:

```sh
MODEL=claude-opus-4-8            # 両サイド一括指定（既定 claude-opus-4-8）
AILEX_MODEL=... PY_MODEL=...      # サイド別指定（指標D: 例 AILEX_MODEL=claude-haiku-4-5 PY_MODEL=claude-opus-4-8）
ROUNDS=3                         # 修復ラウンド上限（両サイド共通）
REPEATS=3                        # 反復回数（分散把握）
```

実行すると起動時にスモークテスト（strict enum + 強制 tool_choice が通るか）を行い、失敗すれば**タスク失敗に化けさせず停止**する。結果は `ailex/logs/compare-<時刻>.json` に保存される。

> ⚠ **compare.ts の Python サイドはモデル生成コードを `python3` サブプロセスで実行する**（HumanEval 等と同じ方式・タイムアウト付き）。ネットワーク隔離した使い捨て環境での実行を推奨。信頼できない環境では実行しないこと。

## 評価指標

- **functional correctness**（モデルに見せない隠しテストの合格）＝主指標
- **pass@1**（初回で合格したか）
- **repair rounds**（合格までの試行回数・両サイド）
- **token / API cost**（$、モデル別単価）
- Python 側のエラー内訳（構文 / 型 / 実行時 / 誤答）＝記述統計

**Ailex 側の型エラー数は主指標にしない**（マスクにより構造的にゼロになる＝循環論法）。
実験の主眼は「制約がモデルサイズや修復コストをどれだけ肩代わりするか」（例: Ailex＋弱いモデル vs Python＋強いモデル）。

## 限界（隠さない）

- 実 LLM の数値は E1–E3（[EXPERIMENTS.md](EXPERIMENTS.md)）のみで、タスク台は10問・簡単。主要な発見はむしろ否定的：Python 側は Haiku でも天井（有名関数は事前知識で解ける）で「制約が規模を肩代わりする」仮説はこの台では判定不能、かつ逐次マスクは入力トークンが7〜9倍かかる。
- タスク台が依然浅い（10問・単純な合成と分岐まで）。多相・再帰・任意定数・データ構造の操作は未対応。
- 数値リテラルは固定パレットのみ（任意定数を出せない）。
- 逐次マスクは一手ごとに API 呼び出しを重ねるため入力トークンが嵩む。等コスト比較の統制が必須。
- 比較は naive な意味では「公平」でない。Ailex は型の足場を設計上もらっている——それを測るのが目的である。
- 単発実行はノイズを含む。反復（`REPEATS`）と分散の報告が要る。
- タスクのテスト設計が結果を左右する（公開例が緩いと過適合が起きる。境界点を隠しテストに必ず入れること）。
- Python 実行はサンドボックス化していない（タイムアウトのみ）。
