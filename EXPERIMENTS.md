# Ailex 実験ログ

実測はすべて実 Anthropic API で実行。生ログは `ailex/logs/` の JSON（コンソール出力は同名 `.txt`）。
数値の再現には `ANTHROPIC_API_KEY`（`ailex/.env` 可）が必要。

## 共通設定

- タスク台: 10問（数値合成5: sq/normsq/cube/norm/hypot、分岐5: max/min/relu/abs/sign）
- 採点: 隠しテスト（モデルに見せない）による機能的正しさ。修復フィードバックは公開例のみ（テストリーク無し）
- ROUNDS=3（両サイド共通の修復ラウンド上限）
- Ailex 側: 逐次マスク生成（strict enum + 強制 tool_choice）。型・構文エラーは構造的に出力不能
- Python 側: 関数丸ごと生成 → python3 実行 → 失敗を差し戻し修復
- 価格: Opus 4.8 $5/$25 per 1M、Haiku 4.5 $1/$5 per 1M（入力/出力）

## 所見（この実測が支持すること／しないこと）

**支持されること**
1. 実 LLM は逐次マスク下でほぼゼロショットの新言語を高確率で正しく埋められる（Opus 100%、Haiku 90%）。型・構文エラーはゼロ（ただしこれはマスクの構造的帰結であり、成果指標ではない）。
2. 修復ループ（構造化フィードバック→再生成）は実モデルで機能する（E2 hypot）。ただし確実ではない（E3）。
3. マスク内でもモデル品質差は残る（Opus 100% vs Haiku 90%）。**制約は well-formedness を消すが、意味の誤りは消さない。**

**支持されないこと（現時点）**
4. 「制約がモデル規模を肩代わりする」（指標D）は**このタスク台では判定不能〜不支持**。Python 側が Haiku でも天井（30/30）であり、有名関数はモデルの事前知識だけで解けてしまう。
5. 等コスト比較では Ailex が一貫して不利（**入力トークン 7〜9倍**）。逐次マスクの「一手ごとに文脈を再送する」構造が原因。

**帰結（次の実験に必要なもの）**
- Python+Haiku が実際に落ちる難度のタスク台。それには言語コアの表現力拡張（リスト操作・再帰など）が前提
- コスト側の改善（複数ホールの一括問い合わせ、文脈の差分送信など）
- `sign` 型の過適合対策：公開例の設計指針（境界点を含める）、または性質ベースの契約の活用

**総コスト**: 3実験合計 約 $0.74

# Q1: 構造化フィードバック vs 自然言語フィードバック（2026-07-10）

**問い**: 検証器の応答を「機械可読な構造化データ（期待型・スコープ候補[名前:型]・失敗例）」として修復ループに還流すると、自然言語フィードバックに比べ pass@k が上がり修復ラウンドが減るか？（PRIOR_ART.md の最有力の新規性候補。Self-Repair(ICLR 2024)は「フィードバックの質が鍵」を示したが NL 止まり。）

**設計**: Ailex 本体式を全項生成（マスク無し）。同一モデル・同一タスクで**フィードバックの形式/内容だけ**を2条件で振る。
- `nl`: コンパイラ/リンタ風の散文（期待型/実際型、失敗例。スコープは列挙しない）
- `structured`: 機械可読 JSON（期待型＋**スコープ候補[名前:型]**＋失敗例）
- regime: `TEACH=min`＝構文は完全に教えるが**使える関数名（スコープ）を伏せる**。これで失敗が「API名の当て違い」層に集中し、スコープ開示の効果を測れる。
- 採点は隠しテスト、修復は公開例のみ（テストリーク無し）。ハーネス: `ailex/q1.ts`。

## E-Q1: Haiku 4.5, TEACH=min, REPEATS=5（n=50）

ログ: `logs/q1-2026-07-10T10-57-09-765Z.json`, `logs/q1-haiku-min-rep5.txt`

| | nl | structured |
|---|---|---|
| pass@1 | 39/50 (78%) | 37/50 (74%) |
| pass@k | **40/50 (80%)** | **49/50 (98%)** |
| 平均rounds | 1.64 | 1.34 |
| コスト | $0.116 | $0.094 |

**タスク別 pass@k（5反復中）**:

| task | nl | structured | 備考 |
|---|---|---|---|
| normsq | **0/5** | **5/5** | 正解に `dot` が必要 |
| norm | **0/5** | **4/5** | 正解に `dot` が必要 |
| 他8 (sq/cube/hypot/max/min/relu/abs/sign) | 5/5 | 5/5 | 引き分け |

## 所見（何が言えて、何が言えないか）

**言えること（機構が明快）**:
- 効果は「正解に**非自明なAPI名**（`dot`）が要るタスク」に集中。そこでは structured が 9/10 回復、nl は 0/10。API名が当てやすい（`mul/add/sqrt/if`）タスクでは差なし。
- pass@1 はほぼ同じ（78% vs 74%）＝差は**生成でなくフィードバック**由来。round-1 はフィードバックを使わないため当然。
- 機構: nl は「型が合いません／関数でない」と言うだけで**何が使えるか**を明かさない→モデルは名前を当て続けて発散。structured はスコープ[名前:型]を明かす→1手で回復。
- **結論（条件付き・肯定的）**: 構造化フィードバックは、回復に「推測不能なAPI知識」が要る場面で、名前当ての発散ループを一手修正に変える。これは Self-Repair(NL)が捉えず、スコープを構造化データで還流した先行例も見当たらない領域。

**言えないこと（正直な限界）**:
- 効果は 10 タスク中 **2 タスク**（ともに `dot` 依存）が全て。「pass@k +18pt」は**条件付き効果**であり、全タスク一律の底上げではない。
- 単一モデル（Haiku）・単一 regime（スコープ秘匿）。regime はやや人工的だが、「新言語のAPIをモデルが知らない」というコールドスタントの忠実なモデル化ではある。
- タスク台が浅い（数値・ベクトル）。より広いタスクで効果の一般性は未検証。
- 「差の内容 vs 形式」は未分離: structured の勝ちは**スコープという情報を足したから**か、**JSON という形式だから**か。次段で NL-rich（同じスコープ情報を散文で渡す）群を足せば分離できる。

**次の一手**: (1) NL-rich 群を足して「情報 vs 形式」を分離、(2) 他モデル（Sonnet/Opus）で効果が縮むか（強モデルほど推測が上手く効果が減る仮説）、(3) `dot` 以外の「推測不能API」タスクを増やして効果の再現性を確認。

## E-Q1b: 交絡分離（内容 vs 形式）Haiku, TEACH=min, REPEATS=5, 3群（n=50/群）

ログ: `logs/q1-2026-07-10T11-07-38-018Z.json`, `logs/q1-3arm-rep5.txt`

新群 `nl_rich`＝structured と**同じ内容（期待型＋スコープ候補[名前:型]＋失敗例）を散文で**渡す。
これで nl→nl_rich が「内容（スコープ情報）効果」、nl_rich→structured が「形式（JSON）効果」を分離する。

| arm | pass@k | 平均rounds |
|---|---|---|
| nl（誤りのみ・散文） | 40/50 (80%) | 1.66 |
| nl_rich（スコープを散文で列挙） | **50/50 (100%)** | 1.26 |
| structured（同じ内容を JSON で） | **50/50 (100%)** | 1.34 |

- **内容効果 (nl → nl_rich): Δ+10**（スコープ情報を足すと 80%→100%）
- **形式効果 (nl_rich → structured): Δ+0**（同じ情報なら JSON も散文も同じ）

### 結論（この分離実験が確定させたこと）

1. **効くのは「スコープ／型情報の開示」であって「機械可読な構造化形式」ではない。** 散文でスコープを列挙すれば JSON と同等（両方 pass@k 100%）。
2. → **Ailex 中核 #2 の「検証器応答を*機械可読な構造化データ*として還流する」という新規性の枠は、この実験では支持されない。** 価値は「何が使えるか（スコープ＋型）を明かすこと」にあり、形式は無関係。
3. これは PRIOR_ART の Hazel/ChatLSP（arXiv:2409.00921、型・型文脈を LLM に与えると 5%→60%）と**整合**する。Hazel も型/スコープ文脈を（JSON でなく）プロンプト文脈として与えている。つまり本実験は「型/スコープ情報が効く」という既知の結果を修復ループ設定で再確認したもので、**"構造化形式" 部分に固有の新規性は無い**。

### 研究上の含意（新規性の再配置）

- 「構造化フィードバック（形式）」は新規性の芯として弱い（形式効果ゼロ・かつ内容効果は Hazel と重なる）。
- 残る差別化があるとすれば「**修復ループ内で**スコープ/型を還流する」点（Hazel は初期文脈注入）だが、これは細い差分で、単独の新規性主張には弱い。
- 実質的な発見は肯定・否定の両面: (肯定) 検証器がスコープ/型を明かすと、未知APIの名前当て発散が一手修正に変わる（実測で明快）。(否定) その価値は「機械可読形式」ではなく「情報内容」にある。

# A: AI生成実験（新コア v0.3.x・2026-07-10〜11）

**問い**: 育った Ailex（レコード・HOF・エイリアス込み）を、モデルは「早見表（約20行）だけ」の in-context 学習でどれだけ書けるか。
**方式**: 8タスクの関数本体式を書かせる（`ailex/core/aigen.ts`）。修復フィードバックは言語ネイティブの構造化診断＋公開 eg。採点は隠しテスト。ROUNDS=3。

## A1: v0.3.1（ラムダは型注釈必須）

| モデル | pass@1 | pass@k | 平均rounds | 修復要因 | コスト |
|---|---|---|---|---|---|
| Haiku 4.5 | 4/8 (50%) | 8/8 (100%) | 1.50 | parse×4 | $0.020 |
| Opus 4.8 | 5/8 (63%) | 8/8 (100%) | ~1.4 | parse×3 | $0.080 |

- 失敗は**全て parse**で、原因はほぼ単一：モデルが JS/Python の習慣で**型注釈なしラムダ** `fn (acc, x) => ...` を書く。
- 意味はほぼ常に正しい（修復1回で全問回復）。**モデルの事前分布と言語の構文が衝突する箇所が、そのまま pass@1 の損失になる**。

## A2: v0.3.2（ラムダの型注釈を任意化＝期待型から推論）——言語側を直して再測定

修正: check モードのラムダは期待関数型からパラメータ型を推論（注釈があれば照合）。conformance 59/59。

| モデル | pass@1 | pass@k | 平均rounds | 修復要因 | コスト |
|---|---|---|---|---|---|
| Haiku 4.5 | **8/8 (100%)** | 8/8 (100%) | 1.00 | なし | $0.013 |
| Opus 4.8 | **8/8 (100%)** | 8/8 (100%) | 1.00 | なし | $0.057 |

- 生成物を確認：Haiku は 6/8、Opus は 5/8 のタスクで**実際に注釈なしラムダを使用**（推論経路が本当に使われた）。
- コストも約30%減（修復ラウンド消滅のため）。

## 所見

1. **測る→言語を直す→再測定、のループが閉じた初の実例。** 「モデルが自然に書く形を言語が受け入れる」方向の1修正で、pass@1 が 50–63% → 100% に。
2. 教訓の一般形：**AI向け言語の構文は、モデルの事前分布（既存言語の習慣）と喧嘩しないことが第一級の設計制約**。中置演算子の採用（Q1教訓）と同型の判断。
3. 正直な限界: 8タスク・REPEATS=1・タスクは中級まで。A1→A2 は早見表も1行変わっており（新機能の記載）、純粋な言語効果と primer 効果は完全には分離していない。ただし A1 の失敗が全て「注釈なしラムダの parse エラー」だったことから、主因が言語修正であることは強く支持される。

## A3: 計測器の拡張（16タスク・v0.5.1・2026-07-13）

受け入れテストが v0.3.2 以降満点で飽和し、v0.4/v0.5 の新機能（Option・レコード・文字列）が未計測だったため、
8→16タスクに拡張（safeDiv/sumParsed/nameFirstBig/initials/moveBy/countCat/labelAll/bestScore を追加。早見表も v0.5.1 版に更新）。

| モデル | pass@1 | 失敗タスク | 修復要因 | コスト |
|---|---|---|---|---|
| Haiku 4.5 | 15/16 (94%) | nameFirstBig | type×2, parse×1 | $0.033 |
| Opus 4.8 | 15/16 (94%) | nameFirstBig | type×3 | $0.158 |

**決定的な観察**: 両モデルの唯一の失敗が**同一タスク・同一原因**。両方とも
`unwrapOr(map(find(es, fn (e) => e.amount > lim), fn (e) => e.cat), "none")`
——つまり **`map` が Option に効くと確信していた**（Rust の `opt.map` / Haskell fmap の事前分布）。
3ラウンドの型フィードバックでも「番兵レコードで覗く」正解形（`unwrapOr(find(...), {dummy}).cat`）には辿り着かず、モデルは不自然な形を書くことを事実上拒否した。

他は全問1発: Option の基本（safeDiv の `if(b == 0, none, some(a / b))`）、fold+none（bestScore）、parse+集計（sumParsed）、レコード再構築（moveBy）、文字列パイプライン（initials）まで、早見表だけで書けている。

## A4: v0.5.2（map を Option に多重定義）——言語側を直して再測定

修正: `map(o, f) : (Option[T], (T)->U) -> Option[U]`（A3 で両モデルが書いた形をそのまま合法化）。conformance 89/89（A3 の形そのままのケース含む）。

- A4 の nameFirstBig で両モデルが1手目に書いた式は A3 と**一字一句同じ**——今度はそれが正解になった。

## 所見（A3→A4）

1. **「測る→言語を直す→再測定」ループの2周目が閉じた**（1周目は A1→A2 のラムダ注釈）。今回も「モデルの事前分布に言語を合わせる」1修正で 94%→100%。
2. パターン化した教訓: **モデルが繰り返し書く不正な形は、モデルの誤りではなく言語の欠落を示す測定値**。A1（注釈なしラムダ）も A3（map-over-Option）も、修正後はモデルの自然な1手目がそのまま正解になった。
3. match 導入の優先度は当面後退（Option の map で番兵ピークの主要因が消えた）。将来の再評価対象として保持。
4. 正直な限界: A3→A4 も早見表が1行変わっており（map の Option 対応の記載）、言語効果と primer 効果は完全には分離していない。ただし A3 の失敗が全て map-over-Option の型エラーだったことから、主因が言語修正であることは強く支持される。REPEATS=1。
