# Ailex v0.1 dogfood — 設計の痛点（実証）

> 実在の言語の手順(PROCESS.md Step 3)に従い、v0.1 で非自明なプログラムを実際に書いて走らせ、
> 設計の痛点を推測でなく実証で洗い出した。証拠は `examples/dogfood.ax` と各 /tmp 実験。

## 良かった点（v0.1 で素直に書けた）

`examples/dogfood.ax`（eg 9件すべて達成・`main()` 実行OK）:
- 統計 `mean` / `variance` / `stddev`（`sum`/`dot`/`toFloat`/`length` と `let` で書ける）
- `clamp`（nested if）、`grade : Int -> String`（if で String を返す）、`lerp`
- **中置演算子＋`let`＋nested if** で「普通のスカラ/リスト処理」は快適。契約 `eg` が実行され安心。診断が構造化。

## 予想外に「動いた」もの（SPEC の想定と違う）

- **再帰が動く**（両バックエンド）: `fact(5)=120`。実装は全関数を globals に入れ eval が解決するため自然に動く。
  → **SPEC の「v0.1 非再帰」は誤り。再帰はサポート**（ただし停止性は保証しない＝無限再帰は落ちる）。conformance に固定済。
- **前方参照が効く**: 後で定義した関数を呼べる。
  → **SPEC §1「宣言は使用に先行（依存順）」は実装で強制されていない**（全関数を先に登録）。conformance に固定済。

## 痛点（v0.2 の優先課題・証拠つき）

### C【解決済 ✅】高階関数がバックエンド不一致（健全性バグ）
> **解決（2026-07-10）**: 型検査で関数値の使用を禁止（関数型の引数・戻り値・let、および関数名を値として使うこと）→
> 両バックエンドが一致（実行前に `higher_order_unsupported` で拒否）。conformance に3ケース追加・33/33緑。正規プログラムに誤検出なし。
> v0.2 でラムダ＋第一級関数として正式サポートする際に、この禁止を解除する。

- `apply2(f : (Float)->Float, x : Float)` を型検査は **ok** にし、**JSバックエンドは `apply2(dbl,3.0)=6`** を返すが、
  **インタプリタは「未束縛 'dbl'」で落ちる**（eval の var は env のみ参照、app.fn は文字列で関数値を呼べない）。
- **型が通るのに片方の実行系で落ちる＝不健全。** conformance に HOF ケースが無く見逃していた。
- 対処の二択:
  - (1) **HOF を正式サポート**: インタプリタに第一級関数値を入れ、app が「式を頭に」取れるよう拡張＋ラムダ構文（下の D）。
  - (2) **v0.1 では関数型パラメータを型検査で禁止**して両バックエンドを揃え、HOF は v0.2 へ。
  - 堅い順序: まず (2) で揃え、v0.2 で (1)。

### D【解決済 ✅】ラムダ/無名関数が無い
> **解決（v0.2, 2026-07-10）**: 無名関数 `fn (x : T) => e` を追加。第一級関数値（インタプリタ・JS 両対応）。C の禁止を解除し HOF を正式サポート。

### E【解決済 ✅】`map`/`fold` が stdlib に無い
> **解決（v0.2）**: `map : (List[T], (T)->U) -> List[U]`、`filter : (List[T], (T)->Bool) -> List[T]`、`fold : (List[T], U, (U,T)->U) -> U` を追加。
> `examples/lists.ax` で `scale`/`countPositive`/`product`/`sumSquares` が書けて走る（eg 4件達成）。以前「書けなかった」リスト変換が解消。

### F【解決済 ✅】`Record`/タプルが無い
> **解決（v0.3, 2026-07-10）**: 構造的レコードを追加。型 `{x : Float, y : Float}`・リテラル `{x = 1.0, y = 2.0}`・アクセス `p.x`（連鎖可）。
> `examples/points.ax` で `dist(p, q)`・`centroid`（レコードのリスト＋map）・`inCircle`（ネスト）が書けて走る（eg 4件達成）。
> 未知フィールドの診断は使えるフィールド一覧を開示（Q1 の知見と一貫）。あわせて eg の判定を深い等価に修正。conformance 46/46。

### G 契約の float リテラルが不便（軽微）
- `variance` の eg 期待値が割り切れないと `2.6666666666666665` のような長い literal を書く羽目。
  性質ベースの `ensures` を厚くするか、`eg` に許容誤差指定があると楽（今は厳密値＋1e-9 固定）。

## v0.2 の優先順位（この dogfood が決めた）

1. **C を解決（健全性）** — まず関数型パラメータを禁止して両バックエンドを揃える（最小・即）。
2. **D ラムダ ＋ E `map`/`fold`** — リスト処理の実用性を一段上げる。ここで HOF を正式サポートし C を昇格。
3. **F `Record`** — 構造データ。
4. **A/B の SPEC 訂正**（再帰サポート明記／依存順は非強制と明記）。★本コミットで SPEC 修正済。
5. conformance に HOF・再帰・前方参照を追加（★再帰・前方参照は追加済、HOF は正式化時に）。

## 一言

v0.1 は「スカラ・リストの素直な計算＋分岐＋契約」までは実用的に書けて、走る。
壁は明確: **リスト変換（HOF/ラムダ/map）と構造データ（Record）**、そして**HOFの健全性**。次の一手が証拠で決まった。

# 第2ラウンド（2026-07-11・v0.3 の全機能で家計簿を書く）

題材: `examples/ledger.ax`（支出レコードの集計・カテゴリ絞り込み・最大値・予算超過判定）。レコード＋リスト＋文字列＋HOF＋契約を総動員。結果: 書けた（eg 4件達成）。その過程で2つ発見。

### H【解決済 ✅・意味論バグ】複合値の == が参照比較だった
- `[1.0, 2.0] == [1.0, 2.0]` が **false**、`{x=1.0} == {x=1.0}` も false。型検査は通るのに値が静かに間違う、最も危ない類のバグ（/tmp/eq.ax で実証）。
- **修正**: `==`/`!=` を構造等価に（`structEq`: 数値は厳密・リスト/レコードは深い比較・関数は参照）。JS バックエンドにも `$rt.eq` を入れて両者の意味論を一致させた。conformance に3ケース追加。
- 契約 `eg` の判定（`valEq`・数値許容誤差つき）とは意図的に区別: 契約はテスト断言（許容）、`==` は言語の演算子（厳密）。

### I【解決済 ✅・人間工学】型エイリアスが無い
- 家計簿でレコード型 `{name : String, amount : Float, cat : String}` を**8回**書かされた（シグネチャ＋ラムダ引数で毎回）。
- **修正**: `type Expense = {name : String, ...}` 宣言を追加（透明エイリアス・宣言は使用に先行・alias-to-alias 可）。ledger.ax を書き直して大幅に短縮。AI 生成でもトークン削減に直結する。
- conformance に3ケース追加（シグネチャ＋ラムダでの使用／連鎖／未宣言はパースエラー）。

### 残っている軽微な観察
- レコードの部分更新は全フィールド再構築で書ける（`{x = p.x + dx, y = p.y}`）。冗長だがフィールド数が少ないうちは許容。関数型更新構文 `{p | x = ...}` は将来の候補。
- G（契約 float リテラルの不便）は未対応のまま。

conformance: 46 → **52/52 緑**。全例（demo/dogfood/lists/points/ledger）回帰なし。

# 第3ラウンド（2026-07-11・テキスト処理を書こうとする → v0.4）

方法: リリース前に最も薄い領域（文字列・安全な失敗）を4つの probe で突いた。

## 発見（すべて実証・すべて解決 ✅）

### J【解決済 ✅】文字列 stdlib が薄すぎる
- `toString`（数値→文字列）・`split`・`contains` が無く、レポート整形・単語処理が一切書けなかった（probe 1/2/4 が not_a_function）。
- **修正**: `split/join/contains/substring/trim`（具体型）＋ `toString`（多相: Int|Float|Bool|String→String、関数は型検査で拒否）を両バックエンドに追加。

### K【解決済 ✅】安全な既定値つき取得が無い
- `head([])` は実行時エラーのみ（Option も既定値変種も無い）。
- **修正**: `headOr(v, default)` / `getOr(v, i, default)` を追加（非破壊的・head/get は従来どおり）。完全な Option[T] は v0.5 以降の候補。

### L【解決済 ✅・CLI バグ】実行時エラーで CLI がクラッシュ
- probe 3: `main()` 内の `head([])` で **Node のスタックトレースを吐いて死んだ**。言語の看板（構造化診断）に反する。
- **修正**: `run` の main() 実行を捕捉し、構造化診断 `{code: "runtime", at: "main()", detail}` を返して exit 1。

## 実証プログラム
`examples/words.ax`（wordCount / longest / slug / report、eg 4件達成）。`longest` は注釈なしラムダを fold で自然に使用（v0.3.2 の推論が実コードで機能）。

conformance: 59 → **67/67 緑**。全例回帰なし。

# 第4ラウンド（2026-07-12・「不在」を書こうとする → v0.5）

probe 2本: 文字列→数値の変換（M）と「条件で最初の1つを探す」（N）。

### M【解決済 ✅】文字列→数値が一切書けない
- `parseInt` が存在しない（not_a_function）。テキスト入力の処理が根本的に不可能だった。
- **修正**: `parseInt(s) -> Option[Int]`・`parseFloat(s) -> Option[Float]`（失敗は none）。

### N【解決済 ✅】find が無く番兵値が要る
- `filter`＋`headOr` で書けるが、既定値に**番兵レコード** `{amount = -1.0, cat = "NONE"}` が必要＝Option が発明された理由そのもののバグ源。
- **修正**: `Option[T]` を型として導入。`some/none/isSome/unwrapOr`、`find(v, 述語) -> Option[T]`。
- `none` は check 位置でのみ型が付く（双方向設計と一貫）。Option 値の契約 `eg f([]) = none` も動く。

実証: `examples/option.ax`（firstBig/readScore/sumScores、eg 6件達成）。
conformance 67 → **78/78 緑**。全7例回帰なし。受け入れ（aigen Haiku）8/8 pass@1 退行なし。

# 第5ラウンド（2026-07-12・成績表＝複数関数の実プログラム → v0.5.1）

題材: `examples/gradebook.ax`（行テキスト解析→Option で不正行を弾く→集計→レポート整形。6関数）。
**書き手として本物の痛点に当たった**——そして重大なバックエンド不一致も1件検出。

### O【解決済 ✅】fold の初期値に [] / none が書けない（双方向伝播の不足）
- 自然な `fold(lines, [], ...)` が「空リスト(型不定)」で拒否され、`fold(ss, none, ...)` も同様。巻き添えでラムダが「注釈必須」に落ちる。
- **修正**: check 位置の fold は期待型を初期値へ伝播（`[]`・`none` に型が付き、ラムダの acc 型も決まる）。`[]` の check 規則も追加（引数位置・契約でも型が付く）。

### P【解決済 ✅】concat の4重ネスト（文字列連結の人間工学）
- レポート整形が `concat(a, concat(b, concat(c, d)))`。型は通るが書くのも読むのも苦痛。
- **修正**: `+` を String の連結に許可（`-`/数値混在は従来どおり拒否）。モデルの事前分布（JS/Python）とも一致。

### Q【解決済 ✅・重要】JS バックエンドの parseInt が常に none
- **eg（インタプリタ）は9件全部通るのに `main()`（JS）だけ「0 students」**——dogfood がバックエンド不一致を検出。
- 原因: tojs の PRELUDE はテンプレートリテラルで、正規表現の `\d` が文字列化で `d` に化けていた。
- **修正**: `\\d` にエスケープ。**教訓: contracts はインタプリタのみ・evals は両バックエンドを通る。新しい stdlib には必ず evals を併記**（conformance に JS 経路の evals を追加して固定）。

### 収穫（バグでなく確認できたこと）
- `eg` 契約が**書き手（私）の実バグ**（`, top` のカンマ抜け）を捕まえた——契約が働く実例。
- 「isSome+unwrapOr で覗くために番兵既定値が要る」（best 関数の `unwrapOr(acc, s).score`）は残る不便。**パターンマッチ（match）の将来根拠**として記録。

conformance: 78 → **85/85 緑**。全8例緑。受け入れ（aigen Haiku）8/8 pass@1 退行なし。
