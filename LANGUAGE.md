# Ailex 言語設計 v0.1 —— AI が書くための中間言語

> 立ち位置: Go / Python のような「実在の、よく設計された、使える言語」を作る。
> ただし書き手は人間でなく **LLM**、読み手が人間。哲学([Ailex.md](../Ailex.md))でなく、
> **実装可能な設計決定**をここに固める。決定は opinionated（一つの道）。異論あれば覆す。

---

## 0. 何を作るか / 成功の定義

- **AI中間言語**: AI が生成・検証し、実行時は host ランタイムへ落とす中間表現。人間向け高級言語ではない。
- **成功 = 世界初でなく「使える・よく設計されている」。** Go/Python と同じ基準。research novelty は狙わない。
- ここまでの実測（[EXPERIMENTS.md](EXPERIMENTS.md)）は novelty の証明でなく **設計の根拠** として使う。

---

## 1. 設計の柱（各々、実測 or Go/Python の教訓に紐づく）

1. **正規形は一つ（one obvious way）。** 意味が等しい2プログラムは L1 で byte 一致。スタイルというエントロピー源をモデルから消す。← gofmt の全域化 / E1–E3。
2. **文脈に収まる小ささ。** 言語コア＋stdlib 要約が LLM のプロンプトに丸ごと入る規模を保つ（数千トークン）。corpus-zero 問題への解。← ハードな設計制約。
3. **型は軽量・実利本位。** 安価にエラーを捕まえる型は入れる。重い形式検証は入れない。契約(`ensures`)と実例(`eg`)で意味を押さえる。← E1–E3（制約は well-formed を守るが意味は守らない）。
4. **スコープ開示が第一級。** 「いま何が・どの型で使えるか」をコンパイラが機械的に答える。これが修復で一番効いた。← Q1/Q1b（効いたのはスコープ情報）。
5. **エラーは構造化・局所化。** 診断は英文でなく `{期待型, スコープ候補, 未達義務, 位置}`。← Q1b。
6. **決定的・再現可能。** 同じプログラム→同じ結果。副作用は型に出す（effect 型、v0.2）。← エージェント信頼性。
7. **本物の相互運用。** おもちゃにしない。実行時は実ランタイム（まず JS）へ transpile し、実ライブラリを呼べる。← 「中間言語」の本義。
8. **三層。** L0（内容アドレスされた型付きグラフ＝真実）／L1（一意な正規形＝AIの生成面）／L2（人間向け投影）。編集面は L1。

## 2. 型システム（v0.1）

- 静的・構造的。`Int` `Float` `Bool` `String` `List[T]` `Fun(T..) -> T`。v0.2 で `Record{..}` `Option[T]`。
- **null なし。** 欠如は `Option[T]`（v0.2）。それまで欠如を作らない。
- 双方向型検査（合成 synth / 検査 check）。式は原則 check 位置に置き、型は宣言に前置（単調型付け可能性は「安価な保証」として維持。ETH と機構は同じでよい＝借り物で構わない）。
- 数値は `Int`/`Float` を分離（Python の暗黙昇格はしない＝曖昧さを減らす）。

## 3. 構文（正規形・canonical）

- 式指向。`let x : T = e in e`、関数適用 `f(a, b)`、条件 `if(c, a, b)`、比較 `>=` 等、リテラル、リスト `[a, b]`。
- 関数定義はシグネチャ前置（型・契約・実例が本体に先行）:
  ```
  fn norm (v : List[Float]) -> Float
    ensures ret >= 0.0
    eg norm([3.0, 4.0]) = 5.0
  body Float
    sqrt(dot(v, v))
  end norm
  ```
- **書き方は一通り。** 空白・改行・順序はフォーマッタが一意化（gofmt 相当・違反は正規化されるだけでエラーにしない）。

## 4. 実行モデルと相互運用

- **Ailex は L0 から host へ transpile される中間言語。** v0.1 のターゲットは **JavaScript/TypeScript**（理由: 実装が TS で地続き・npm という実エコシステムに FFI・実行が本物）。
- 将来、同じ L0 から Python / WASM ターゲット（多言語 lowering）。これが「中間」言語たる所以。
- FFI: host 関数を型付きで import（`extern sqrt : (Float) -> Float from "Math"` 的な宣言、v0.2 で具体化）。stdlib の多くは host への薄い型付きラッパで始める。

## 5. 契約と検証（第一級・軽量）

- `requires` / `ensures` / `eg`（実例）を宣言に書く。実例は**実行可能なテスト**として評価される。
- 検証は「型（安価・常時）」＋「契約・実例（実行時）」の二段。重い定理証明は入れない（Dafny にはしない）。
- 失敗は §1-5 の構造化エラーで返す。

## 6. AI 向けツール（第一級）

- **scope サーバ**: 任意位置で「スコープ内の名前と型」を機械可読で返す（Q1 の知見を言語機能に昇格）。生成器/エージェントが常に参照できる。
- **フォーマッタ**: L1 正規形へ一意化。
- **構造化診断**: コンパイラの唯一の出力形式（英文診断は L2 投影として別途生成可）。
- 言語仕様全体が1つのプロンプトに収まる **単一ファイルのミニ仕様** を維持（in-context 学習用）。

## 7. v0.1 の実装範囲（現状との差分）

**既にある**（[prototype.ts](prototype.ts) 他）: パーサ・双方向型検査・評価器・L1/L2・`if`・関数・`Float`/`Vec`・リテラル・契約(`ensures`/`eg`)・スコープ提示・構造化フィードバック。

**v0.1 で足す**（「実際にプログラムが書ける」最小線）:
1. 型: `Int` `String` `Bool`演算 `List[T]`（`Vec Float` を `List[Float]` に一般化）
2. `let ... in`（局所束縛）
3. 複数関数の相互呼び出し（依存順・v0.1 は非再帰、再帰は v0.2）
4. **JS への transpile バックエンド**（現状は TS インタプリタ。実行を「本物」にする）
5. `scope` サーバと `format` を CLI コマンドに
6. 小さな stdlib（算術・リスト基本・比較）を host ラッパで

**やらないこと（v0.1）**: 再帰・多相・Record/Option・effect 型・FFI 宣言構文・パッケージ/モジュール・並行性。すべて v0.2 以降。**小ささを守る**のが最優先。

## 8. 正直な但し書き（規模と採用）

- 言語は大きい。Go/Python は多年＋エコシステム。v0.1 は「非自明なプログラムが書けて本物に実行される」ところまでで、それでも大仕事。
- **採用は別の山。** corpus-zero（訓練データ皆無）は残る。緩和策は §1-2（in-context に収まる小ささ）＋ §4（実言語への transpile で合成コーパスを機械生成）だが、広く使われる保証はない。
- それでも「よく設計され、動く、AIが書ける中間言語」を作ること自体が目的として成立する（あなたの創作言語群と同じ精神）。
