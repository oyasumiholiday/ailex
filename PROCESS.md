# 実在の言語はどう作られたか / Ailex の構築手順

> 出典: [Go (Wikipedia)](https://en.wikipedia.org/wiki/Go_(programming_language))、[Go: A Documentary](https://golang.design/history/)、
> [Crafting Interpreters](https://craftinginterpreters.com/)（実装パイプラインの定番）、[LLVM Kaleidoscope tutorial](https://llvm.org/docs/tutorial/MyFirstLanguageFrontend/LangImpl02.html)。
> 目的: 実装に入る前に「実在の言語がどの順で作られたか」を押さえ、後戻りを減らす。

---

## 1. 実在の言語の実際の道のり（年単位・段階的）

**Go**（設計 2007 → 初期コンパイラ着手 2008/1 → 公開 2009/11 → v1.0 2012/3 → 自己ホスト 1.5 2015）
- 動機は Rob Pike が C++ の遅いコンパイルに苛立ったこと。**小さく尖った設計**（一つの道）。
- **最初のコンパイラは C で書いた**（ブートストラップ問題: Go を作るのに Go が要る）。
- **自己ホスト（Go で Go を書く）は v1.5、つまり最初の実装から7年後**。自動変換ツールで C→Go 変換。Go 1.4 が 1.5 をコンパイルし、以後 N が N+1 をコンパイルする循環。
- 教訓: **自己ホストは最初にやらない。** まず別言語で reference 実装を作り、動くものを育て、ずっと後で self-host。

**Python**（Guido van Rossum, 1989 着手 → 1991 公開）: **C で書いたインタプリタ**から出発。仕様と実装が一体で育った。

**共通パターン**: 着想 →（別言語で）reference 実装 → 小さく動かす → テストで固める → dogfood → 1.0 →（ずっと後で）self-host / 最適化 / エコシステム。**どれも年単位。1.0 は通過点。**

---

## 2. 実装の標準パイプライン

```
ソース → 字句解析(lexer) → 構文解析(parser) → AST
      → 意味解析(型検査・スコープ) → IR → コード生成 or インタプリタ実行
```

- lexer: 文字列 → トークン列。
- parser: トークン → AST（再帰下降＋二項式は演算子優先順位、が定番の組合せ）。
- 意味解析: 型検査・スコープ解決。
- 実行: **tree-walking インタプリタ**（AST を直接評価）か、**コンパイラ**（IR→機械語/別言語）。

---

## 3. 実践の順序の知恵（Crafting Interpreters 流）

1. **まず tree-walking インタプリタ。** 反復が速く、言語設計を早く触れる。最適化・別言語バックエンドは後。
2. **早期に conformance テスト群（golden tests）。** 「入力プログラム→期待する結果/診断」の対で大量に持つ。実在の言語は巨大なテスト群を持ち、それが**安全な言語成長とリファクタを可能にする**。
3. **仕様と reference 実装を一緒に育てる。** 仕様([SPEC.md](SPEC.md))を単一の真実にし、実装をそれに合わせる。
4. **dogfood。** 言語自身で本物のプログラム（stdlib・例）を書き、設計の痛点を体感する。
5. **2つ目の実装（別バックエンド）は、同じテスト群で検証する。** Go の C→Go 変換も「振る舞いが同じ」を担保にした。
6. **self-host / 最適化 / パッケージ / エコシステムは、ずっと後。**

## 4. Ailex の現在地（0→1 の山は越えている）

[prototype.ts](prototype.ts) に既にある: **lexer・parser・AST・双方向型検査・tree-walking インタプリタ・L1/L2・構造化診断**。
→ **標準パイプラインの骨格は完成済み。** Go/Python が最初に数ヶ月かけた部分は既にある。次にやるのは、実在の言語が「その後」やったこと（テスト整備・言語の肉付け・2つ目のバックエンド・dogfood）。

## 5. Ailex の構築手順（この知恵で SPEC §12 を並べ替える）

**現状 prototype.ts は `Vec Float`・`group/sig`・`>=` のみの旧設計。** SPEC v0.1 は新設計。だから:

- **Step 0 — reference 実装を新ファイルに立てる。** 旧 prototype/compare/q1 は実験記録として温存。v0.1 コアは `core/` 等に新規（tree-walking のまま。バックエンドは急がない）。
- **Step 1 — conformance テスト群を最初に用意。** `tests/` に「Ailex ソース → 期待する {型検査結果 / 実行値 / 診断}」の golden 対を置き、`node` で回すランナー。SPEC 付録の例＋各機能の最小例から。**機能を足す前にテストの器を作る。**
- **Step 2 — 言語コアを機能ごとに、テスト付きで育てる**（SPEC の順）:
  1. 型 `Int/Float/Bool/String/List[T]`（`Vec`→`List` 一般化）
  2. 中置演算子＋優先順位パーサ
  3. `let ... in`
  4. stdlib を暗黙スコープ化（`group/sig` 廃止）＋ §8 の組み込み群
  各機能で golden テストを足し、緑を保つ。
- **Step 3 — dogfood。** stdlib の一部や例題を Ailex 自身で書き、設計の痛点を洗う。
- **Step 4 — 2つ目のバックエンド（L0→JS transpile）。** Step 1 の同じテスト群で「インタプリタと JS 実行が一致」を担保。ここで初めて「実行が本物・npm 相互運用」。
- **Step 5 — AI 向けツール（`scope`/`fmt`/`check`/`run` の CLI）を第一級に。**
- **（ずっと後）** v0.2 機能・self-host・エコシステム。

**要点**: 実在の言語の一番の教訓は「**動く reference 実装＋巨大なテスト群を、仕様と一緒に、段階的に**」。Ailex は骨格があるので、**次の一歩は「テストの器（Step 1）」**——ここを飛ばすと言語成長で必ず壊れる。

---

## 6. 規模の現実（正直に）

- Go/Python も 1.0 まで**数年**。個人プロジェクトなら v0.1（非自明なプログラムが本物に動く）でも相応の期間。
- 採用・エコシステムは別の山（corpus-zero 問題も残る）。**まずは「よく設計され、テストで固まり、本物に動く小さな言語」**を作りきることを 1.0 の定義にする。
