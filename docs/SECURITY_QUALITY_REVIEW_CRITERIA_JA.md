# AI 制作で気を付けるセキュリティ/品質チェック

このページは、**AIにWebサイト・業務ツール・Webサービスの安全性と品質をチェックさせるためのレビュー基準**です。特に、非エンジニアが Vibe Coding で作成したサービスを公開する前に、このページをAIへ渡して確認させる使い方を想定しています。

AIが「動くもの」を作れても、**安全・正確・保守可能とは限りません**。対象のコード、設定、クラウド環境、公開URLなど、AIが実際に確認できる情報と一緒にこのページを渡してください。

# AIへのチェック指示

> **このページ自体は編集しないでください。** 各表をレビュー基準として使い、結果は会話または依頼元が指定した別の場所へ回答してください。確認者名、確認日、URL、証拠リンク、承認記録などをこのページへ追記しないでください。
> 
- 対象のコード、設定、環境、公開状態を実際に調べ、推測だけで「問題なし」にしないでください。
- 各項目を **OK / NG / 未確認 / 対象外** のいずれかで判定してください。
- **OK:** 確認できた根拠を、秘密情報そのものを表示せず簡潔に示してください。
- **NG:** 危険な理由、想定される影響、安全な対処方法を示してください。
- **未確認:** 確認できない理由と、追加で必要な情報・設定画面・テストを示してください。
- **対象外:** なぜ対象外なのかを簡潔に説明してください。
- 最初に「🚨 公開停止事項」を確認し、1つでもNGまたは重大な未確認があれば、公開を止めるべき事項として先に報告してください。
- パスワード、APIキー、Cookie、Webhook URL、個人情報などの値を回答へ転載しないでください。
- 個人情報、決済、医療、金融、採用、社内機密、大量のユーザーデータを扱う場合は、AIだけで公開可とせず、専門家レビューが必要と報告してください。
- 修正や外部サービスの設定変更は、依頼者から明示的に頼まれた場合だけ行ってください。

## 推奨する回答順

1. 公開停止事項
2. セクション別のNGと対処方法
3. 未確認事項と追加で必要な情報
4. 対象外の項目
5. 公開前に直す順番

# 🚨 公開停止事項

この表にNGまたは重大な未確認がある場合は、原則として公開を止めてください。

| チェック項目 | ダメな理由 | 対処方法 |
| --- | --- | --- |
| GitHub、ソースコード、HTML、ログ、AIとの会話に秘密情報が含まれていないか | パスワード、APIキー、秘密鍵、DB接続情報、Webhook URLが漏れると、第三者による不正利用や情報流出につながる | コードから削除し、GitHub Secrets、AWS Secrets Manager、Google Cloud Secret Managerなどへ移す |
| 過去に公開した秘密情報を失効・再発行したか | Git履歴やログから文字列を消しても、既に取得された秘密は引き続き悪用できる | 対象の秘密を直ちに無効化し、新しい値を発行して影響範囲を調べる |
| 本番DBがインターネット全体から直接接続できないか | IDとパスワードだけに依存すると、漏洩や総当たり攻撃でDBへ直接侵入される | Private Subnet、VPC、Private IPを使い、アプリや許可済みネットワークだけから接続する |
| Supabaseの公開スキーマでRLSが有効か | RLSがないと、anonキーや利用者の操作から他人の行を読んだり更新できる可能性がある | 全対象テーブルでRLSを有効にし、本人・別ユーザー・匿名ユーザーで各操作をテストする |
| サーバー側で認証と認可を確認しているか | 画面上でボタンを隠すだけでは、APIを直接呼ばれて他人のデータや管理機能を操作される | すべての読取・作成・更新・削除で本人確認、所有者確認、権限確認を行う |
| 管理画面、検証環境、内部ツールが一般公開されていないか | URLを知っているだけで使える状態では、管理操作や未公開情報へアクセスされる | サーバー側の管理者認可に加え、Cloudflare Accessなどで利用者を制限する |
| 問い合わせ、Webhook、アップロードなど外部から受ける入口に防御があるか | 誰でも無制限に呼べる入口は、スパム、不正処理、費用増大、マルウェア配布に悪用される | 署名検証、認証、入力検証、回数制限、Turnstile、ファイル検査を用途に応じて組み合わせる |
| 個人情報、決済、医療、金融、採用、社内機密を専門家が確認したか | AIのレビューだけでは法務、規制、契約、業務固有の重大リスクを判断しきれない | 公開前にエンジニア、セキュリティ、法務など必要な担当者のレビューを受ける |
| バックアップから実際に復元できるか | バックアップが有効でも、壊れている、権限がない、手順が不明などで復旧できない場合がある | 検証環境へ復元し、必要なデータと所要時間を確認する |
| 障害を検知し、以前の状態へ戻せるか | 障害に気づけない、または戻せないと、被害と停止時間が拡大する | 監視・通知を設定し、アプリとDBのロールバック手順を事前に試す |
| AIが重大操作を人の確認なしに実行しないか | 誤判断やPrompt Injectionにより、削除、送金、一斉送信、公開、権限変更が実行される恐れがある | AIの権限を最小化し、取り消しにくい操作の直前に人の承認を必須にする |

# 1. 秘密情報・GitHub・デプロイ

| チェック項目 | ダメな理由 | 対処方法 |
| --- | --- | --- |
| `.env`、秘密鍵、DB接続情報、Webhook URLをGit管理していないか | リポジトリの公開、共同作業者、ログ、バックアップ経由で秘密が漏れる | `.gitignore`で除外し、`.env.example`にはダミー値だけを書く |
| 秘密情報を用途に合った保管場所へ置いているか | チャット、メモ、コードへ分散すると、アクセス制御、監査、ローテーションができない | CI/CDはGitHub Secrets、AWS/GCPはSecret Manager、ホスティングはSecret機能を使う |
| ブラウザへ配信される変数に秘密を入れていないか | JavaScript、HTML、通信内容は利用者のブラウザから確認できる | `NEXT_PUBLIC_`や`VITE_`などは公開情報だけにし、秘密を使う処理はサーバー側へ移す |
| Secret scanningとPush protectionが有効か | 人の目だけでは、誤ってコミットした秘密を見落としやすい | GitHubの検出機能を有効にし、検出を理由なく回避しない |
| 本番、検証、開発のアカウントと秘密を分けているか | 開発中の誤操作や漏洩が、そのまま本番事故につながる | 環境ごとにクラウド、DB、APIキー、Webhook、決済キーを分離する |
| GitHub Actionsのクラウド認証にOIDCを使っているか | 長期間有効なアクセスキーは、漏洩後も悪用され続ける | AWS/GCPとのOIDC信頼を設定し、ジョブごとに短期トークンを発行する |
| GitHub Actionsの権限が必要最小限か | 侵害されたActionや依存関係から、コード改ざんやクラウド操作をされる | ジョブごとに必要な権限だけを与え、環境・ブランチ・クラウドロールを限定する |
| 本番デプロイにレビューとブランチ保護があるか | 誤変更や悪意ある変更が、確認なしで本番へ反映される | 保護ブランチ、必須レビュー、必須テスト、本番環境の承認を設定する |
| 重要な管理アカウントでMFAを有効にしているか | パスワード漏洩だけでGitHub、クラウド、ドメイン、決済を乗っ取られる | 個人アカウントとMFAを使い、共有アカウントを避け、退職者を削除する |

# 2. クラウド・DB・ストレージ

| チェック項目 | ダメな理由 | 対処方法 |
| --- | --- | --- |
| DBポートを`0.0.0.0/0`へ公開していないか | 世界中から接続試行を受け、認証情報漏洩時に即座に侵入される | アプリのSecurity GroupやPrivate Networkだけを許可する |
| 一時的なDB公開が恒久化していないか | 開発のための例外設定は忘れられやすく、本番の弱点として残る | 期限付き、IP限定、TLS必須とし、作業後に閉じる。通常はVPNや踏み台を使う |
| アプリがDB管理者アカウントを使っていないか | アプリの脆弱性がDB全体の削除や権限変更につながる | アプリ専用ユーザーを作り、必要なテーブルと操作だけ許可する |
| DBとストレージを保存時・通信時に暗号化しているか | 盗聴、スナップショット流出、媒体流出でデータを読まれる | 保存時暗号化とTLSを有効にし、暗号鍵の権限も限定する |
| Supabaseの`service_role`キーをブラウザへ渡していないか | `service_role`はRLSを回避できるため、漏れると全データへアクセスされる | サーバーだけに保管し、ブラウザはpublishable/anonキーとRLSを使う |
| Supabase RLSを複数の利用者条件でテストしたか | RLSを有効にしただけでは、広すぎるPolicyや操作別の漏れに気づけない | 匿名、本人、別ユーザー、管理者でSELECT/INSERT/UPDATE/DELETEを試す |
| ストレージとファイル一覧が原則非公開か | URLや一覧から、個人ファイルや未公開ファイルを辿られる | 非公開バケットと署名付きURLを使い、公開物だけ明示的に公開する |
| 連番IDや推測可能なURLだけで他人のデータを取得できないか | URLを書き換えるだけで他人の情報へアクセスされるIDORが発生する | 取得時に所有者と権限を確認し、必要に応じて推測困難なIDを使う |
| 開発・検証環境に本番の個人情報をコピーしていないか | 本番より弱い環境や多くの開発者から個人情報が漏れる | 匿名化データ、マスキング済みデータ、ダミーデータを使う |
| DBマイグレーションをリハーサルしたか | 本番データ量や既存値により、停止、ロック、データ消失が起きる | バックアップを取得し、本番相当データで実行時間と戻し方を確認する |
| 日常作業にrootやOwner権限を使っていないか | アカウント侵害や誤操作の影響がクラウド全体へ広がる | IAMの最小権限、用途別ロール、短期権限昇格を使う |
| 予算アラートと利用上限があるか | AI API、サーバー、ストレージ、メールが暴走すると高額請求につながる | サービス別の予算、クォータ、通知、緊急停止手順を設定する |

# 3. 認証・セッション・認可

| チェック項目 | ダメな理由 | 対処方法 |
| --- | --- | --- |
| 実績ある認証基盤を使っているか | 独自認証は、パスワード、セッション、再設定、攻撃対策の実装漏れが起きやすい | Auth0、Clerk、Cognito、Firebase Auth、Supabase Authなどを優先する |
| パスワードを平文や復号可能な形で保存していないか | DB流出時に利用者のパスワードがそのまま悪用される | 実績あるライブラリでArgon2やbcrypt等の安全なハッシュを使う |
| メール確認、再設定、メール変更を実機で確認したか | 本人確認漏れや再設定の欠陥からアカウントを乗っ取られる | 本人確認済みメール、再認証、通知を組み合わせ、全フローをテストする |
| 再設定・招待・確認リンクが短時間かつ一回限りか | 古いリンクや盗まれたリンクを後から再利用される | 短い有効期限、単回使用、使用後失効を設定する |
| ログインCookieの属性が安全か | Cookie窃取、HTTP送信、CSRFの危険が高まる | `Secure`、`HttpOnly`、適切な`SameSite`、狭いDomainとPathを設定する |
| ログアウトや権限変更後にセッションを失効できるか | 盗まれたトークンや退職者のセッションが使われ続ける | セッションの期限、失効、全端末ログアウト、トークンローテーションを用意する |
| 管理者にMFAと再認証があるか | 管理者アカウントの乗っ取りは全データ・全利用者へ影響する | MFAを必須にし、削除、返金、権限変更などで再認証する |
| 認証と認可を区別しているか | ログイン済みというだけで、他人のデータや管理機能を使えてしまう | APIごとに役割、所有者、組織、対象リソースを確認する |
| 読取・作成・更新・削除のすべてで権限を確認しているか | 一部の操作だけ確認漏れがあると、直接APIを呼ばれて悪用される | 共通の認可処理をサーバー側に置き、操作ごとのテストを追加する |
| 2人の一般ユーザーでID書き換えを試したか | 自分のテストアカウントだけでは他人のデータ漏洩に気づけない | ユーザーAとBでURL、APIのID、ファイルIDを書き換えて拒否されるか試す |
| 管理画面をURLの秘密だけで守っていないか | `/admin`を非表示にしても、推測や検索からアクセスされる | サーバー側の管理者認可とCloudflare Access等を併用する |
| ログインや再設定に回数制限があるか | 総当たり攻撃やメールアドレスの列挙に使われる | IP・アカウント単位のレート制限、共通エラー、異常検知を使う |

# 4. 入力・API・Webの基本防御

| チェック項目 | ダメな理由 | 対処方法 |
| --- | --- | --- |
| 入力をサーバー側でも検証しているか | ブラウザ側の検証は攻撃者が簡単に回避できる | 文字数、形式、範囲、必須、許可URLをサーバーで再検証する |
| SQLを文字列連結で作っていないか | 入力値をSQLとして実行され、閲覧、改ざん、削除につながる | パラメータ化クエリ、プレースホルダー、適切なORMを使う |
| ユーザー入力をHTMLとして直接表示していないか | 悪意あるJavaScriptを実行され、セッションや画面を乗っ取られる | 通常はエスケープし、HTMLが必要な箇所だけ実績あるサニタイザーを使う |
| Cookie認証の変更処理にCSRF対策があるか | ログイン中の利用者に、別サイトから意図しない操作を実行させられる | CSRFトークン、Origin/Referer確認、SameSite Cookieを組み合わせる |
| GETで削除・購入・登録・設定変更をしていないか | リンクの先読み、画像表示、クローラーだけで処理が実行される | 状態変更はPOST/PUT/PATCH/DELETEにし、認証・認可・CSRF対策を行う |
| 認証情報付きAPIでCORSを広く開けていないか | 悪意あるサイトから利用者の権限でAPIを呼ばれる可能性がある | 許可するOriginを限定する。CORSを認証の代わりにしない |
| リダイレクト先を自由入力にしていないか | 正規サイトのURLを使ったフィッシングへ誘導される | 自サイト内パスまたは許可ドメイン一覧だけを受け付ける |
| サーバーが取得するURLを制限しているか | [localhost](http://localhost)、社内システム、クラウドメタデータへ接続されるSSRFが起きる | プロトコルと送信先を許可リスト化し、Private IPやリダイレクトも検査する |
| エラーへ内部情報を表示していないか | SQL、パス、スタックトレース、秘密が攻撃の手掛かりになる | 利用者には一般的なエラーを返し、詳細は秘密を除いた内部ログへ送る |
| 秘密情報や個人情報をURLへ入れていないか | URLは履歴、アクセスログ、解析、Refererへ残る | リクエスト本文や安全なCookieを使い、ログでもマスキングする |
| HTTPSと必要なセキュリティヘッダーがあるか | 盗聴、改ざん、クリックジャッキング、XSSの影響が大きくなる | HTTPS強制、HSTS、CSP、`frame-ancestors`、`nosniff`を用途に応じて設定する |
| 利用者別ページを共有キャッシュしていないか | 別の利用者へ個人情報や認証後画面を返す恐れがある | `Cache-Control: private/no-store`等を使い、公開静的資産だけをCDNキャッシュする |
| 同じ処理の再送で二重実行されないか | 連打やリトライで二重登録、二重メール、二重請求が起きる | 冪等キー、一意制約、処理済みID、トランザクションを使う |
| 重要APIに回数制限があるか | 総当たり、スパム、DoS、費用増大に悪用される | IP、ユーザー、APIキー単位のレート制限と異常検知を設定する |

# 5. 問い合わせ・Slack・Webhook・外部連携

| チェック項目 | ダメな理由 | 対処方法 |
| --- | --- | --- |
| Slack通知のためだけに公開中継APIを作っていないか | 誰でも通知を大量送信できるスパムの入口になる | サーバーからIncoming WebhookまたはSlack APIを直接呼ぶ |
| Slack Webhook URLやBot Tokenを秘密として保管しているか | URLやTokenだけでチャンネル投稿やAPI操作をされる | Secret Manager等へ保存し、ブラウザ、GitHub、ログへ出さない |
| Slackからの受信で署名と時刻を検証しているか | Slackを装った偽リクエストや過去リクエストの再送を受け入れる | raw body、`X-Slack-Signature`、timestampを公式手順で検証する |
| 他サービスのWebhookでも公式の署名を検証しているか | JSONに書かれた送信元名だけでは本物か判断できない | Stripe、GitHub、microCMS等の公式ライブラリや手順で署名を検証する |
| Webhookの重複配信へ対応しているか | 外部サービスの再送により、通知、更新、請求が複数回実行される | イベントIDを保存し、処理済みイベントを再実行しない |
| TurnstileやreCAPTCHAをサーバー側で検証しているか | 画面にウィジェットを置くだけでは、攻撃者が直接APIを呼べる | 送信ごとにサーバーから検証APIを呼び、失敗・期限切れ・再利用を拒否する |
| フォームに回数制限と入力検証があるか | CAPTCHAだけでは人力スパム、API攻撃、費用攻撃を防ぎきれない | IP・利用者単位の制限、文字数制限、送信上限、監視を組み合わせる |
| 社内限定画面をCloudflare Access等で制限しているか | 未公開画面や検証環境が検索・推測から一般公開される | 許可メール、グループ、MFA、端末条件を設定する |
| 外部APIキーの権限と利用元を制限しているか | 漏洩時に任意の場所から高権限操作や大量利用をされる | 最小権限、ドメイン/IP制限、予算、アラート、ローテーションを設定する |
| 外部サービス停止時に無限再試行しないか | 大量重複、キュー詰まり、費用増大、障害連鎖が起きる | タイムアウト、指数バックオフ、最大回数、失敗キューを設定する |

# 6. ファイルアップロード

| チェック項目 | ダメな理由 | 対処方法 |
| --- | --- | --- |
| 拡張子、実MIME、サイズ、枚数をサーバーで制限しているか | 偽装ファイル、巨大ファイル、保存容量攻撃を受ける | 許可リストを使い、内容からMIMEを判定し、上限を強制する |
| 元のファイル名をそのまま保存していないか | パス操作、上書き、危険な文字、個人情報を含む名前の公開が起きる | サーバー側でランダムな保存名を発行し、表示名と分ける |
| HTML、JavaScript、SVG、実行ファイルを不要に許可していないか | 配信時にスクリプト実行、フィッシング、マルウェア配布へ悪用される | 業務上必要な形式だけ許可し、危険形式は拒否または安全な形式へ変換する |
| 画像を読み直し、必要な検査をしているか | 拡張子だけ画像にした悪意あるファイルや脆弱なメタデータを受け入れる | 画像ライブラリで再デコード・再エンコードし、メタデータ除去とスキャンを行う |
| アップロード先をアプリ本体から分離しているか | アップロードしたファイルがサーバー上で実行される恐れがある | 実行権限のないストレージと専用配信ドメインを使う |
| 他人のファイルを閲覧・上書き・削除できないか | IDやURLの推測だけで個人ファイルが漏洩・改ざんされる | 操作ごとに所有者と権限を確認し、署名付きURLを使う |
| 容量、保存期間、削除、通報の運用があるか | 無制限保存、違法コンテンツ、退会後データ残存が問題になる | ユーザー別上限、ライフサイクル削除、通報・削除手順を決める |

# 7. 依存関係・ライブラリ・サプライチェーン

| チェック項目 | ダメな理由 | 対処方法 |
| --- | --- | --- |
| AIが提案したパッケージの実在と提供元を確認したか | 存在しない名前や似た偽パッケージをインストールする恐れがある | 公式サイト、提供者、更新日、利用実績、リポジトリを確認する |
| lockfileを保存しているか | 環境ごとに異なるバージョンが入り、再現不能や突然の不具合が起きる | `package-lock.json`、`pnpm-lock.yaml`等をコミットし、固定インストールする |
| Critical/Highの脆弱性を放置していないか | 公開済みの攻撃方法から侵入される可能性がある | 依存関係スキャンを実行し、更新、置換、削除、影響確認を行う |
| 定期更新の担当と通知があるか | 古い依存関係が長期間残り、脆弱性と互換性問題が蓄積する | DependabotやRenovateを有効にし、定期的に更新する |
| 不要なライブラリと外部スクリプトを削除したか | 攻撃面、通信量、表示速度、外部サービス依存が増える | 利用状況を確認し、不要なSDK、解析タグ、UIライブラリを削除する |
| GitHub Actionsが信頼でき、固定されているか | 侵害やタグ差し替えにより、CIで秘密を盗まれる | 信頼できる提供元を選び、可能ならコミットSHAで固定する |
| テスト・ビルド・型・静的解析がデプロイ前に動くか | 壊れた変更や危険な変更が、そのまま本番へ出る | CIを必須チェックにし、失敗時はデプロイを停止する |
| OSS、画像、フォント、文章のライセンスを確認したか | 利用条件違反、公開停止、損害請求につながる | ライセンス、帰属表示、商用利用、再配布条件を確認する |

# 8. AI特有のリスク

| チェック項目 | ダメな理由 | 対処方法 |
| --- | --- | --- |
| 機密情報や個人情報をAIへ貼り付けていないか | AI事業者の保存、学習、ログ、管理者閲覧を通じて漏洩する恐れがある | 匿名化・マスキングし、企業向け設定や利用許可済み環境だけを使う |
| AIサービスの保存・学習・削除条件を確認したか | 想定外の期間や目的で入力データが保持・利用される | 利用規約、管理者設定、保存期間、学習除外、リージョン、削除方法を確認する |
| AI生成コードを公式資料とテストで確認したか | 古い仕様、存在しないAPI、脆弱な実装を自信ありげに生成する | 公式ドキュメント、テスト、実アプリの動作を突き合わせる |
| AIの出力を信頼できない入力として扱っているか | 生成HTML、SQL、コマンドを直接使うとXSSやデータ破壊につながる | 検証、エスケープ、許可リスト、権限分離を行ってから利用する |
| 外部文書からのPrompt Injectionを想定しているか | Web、PDF、メール内の命令に従い、情報漏洩や不正操作を行う恐れがある | 外部文章を命令ではなくデータとして扱い、ツール実行前にポリシーと人の確認を通す |
| AIへ与えるツールと権限が最小限か | AIの誤判断や侵害が本番全体へ広がる | 読取と書込、本番と検証、テナントごとに権限を分離する |
| 重大操作の直前に人の承認があるか | 削除、送信、公開、購入、権限変更は取り消しが難しい | 対象、件数、金額、影響を表示し、明示的な承認後に実行する |
| 利用者間の会話・ファイル・検索結果が分離されているか | ユーザーAの情報がユーザーBの回答へ混ざる | テナントIDと権限を全検索・保存・キャッシュへ適用し、複数ユーザーでテストする |
| AI出力の事実、数値、引用を人が確認しているか | ハルシネーションにより誤情報、架空の出典、誤判断が公開される | 一次資料と照合し、高リスク分野は専門家が承認する |
| AI APIの利用量と費用に上限があるか | 長大入力、ループ、大量リクエストでサービス停止や高額請求が起きる | 入力長、出力長、回数、同時実行、日額予算を制限する |
| AI機能の悪用対応があるか | 禁止コンテンツ、詐欺、嫌がらせ、自動攻撃にサービスを使われる | 利用規約、入力制限、出力フィルタ、通報、停止、監査ログを用意する |
| AI自身の自己評価だけで安全と判断していないか | 同じAIは自分の見落としや前提誤りに気づけないことがある | 別のレビュー、テスト、公式資料、必要に応じた専門家確認を行う |

# 9. ログ・監視・プライバシー・運用

| チェック項目 | ダメな理由 | 対処方法 |
| --- | --- | --- |
| エラー、認証失敗、権限拒否、重要操作を追跡できるか | 障害や攻撃が起きても発見・原因調査ができない | 必要なイベント、時刻、対象、結果、追跡IDを構造化ログへ残す |
| ログに秘密情報を出していないか | ログ閲覧者、外部監視、保存先から秘密が漏れる | パスワード、Token、Cookie、カード、Webhook URLをマスキング・除外する |
| 個人情報のログを最小化しているか | 不要な個人情報が長期間残り、漏洩時の被害が増える | 必要項目だけ保存し、閲覧権限、保存期間、自動削除を設定する |
| 重大な異常を通知できるか | 5xx増加、攻撃、ジョブ失敗、容量不足、費用急増へ気づけない | 閾値と通知先を設定し、担当者が対応できる内容にする |
| アラートが実際に届くか試したか | 設定ミス、期限切れ、通知先変更により本番時だけ届かない | テスト障害やテスト通知を発生させ、受信と一次対応を確認する |
| DBと重要ファイルを自動バックアップしているか | 誤削除、障害、ランサムウェアで復旧できない | 世代管理、保存期間、別環境保管、削除保護を設定する |
| バックアップから復元を試したか | バックアップが壊れている、暗号鍵がない、手順が不明な場合がある | 定期的に検証環境へ復元し、完全性と所要時間を確認する |
| インシデント対応手順があるか | 初動が遅れ、秘密漏洩や利用者被害が拡大する | 連絡先、停止、失効、調査、告知、再発防止の流れを決める |
| アプリとDBのロールバック方針があるか | 新バージョンで障害が起きても安全に戻せない | 直前アプリへ戻す手順と、後方互換なDB変更を用意する |
| 定期処理やキューが再実行に耐えるか | 再試行により二重メール、二重更新、二重請求が起きる | 処理ID、一意制約、チェックポイント、冪等処理を使う |
| 退会、削除、出力、同意撤回が機能するか | 利用者のデータが残り続け、説明と実態が一致しない | データ範囲、保管義務、削除期限を決め、実際にテストする |
| ドメイン、クラウド、DB、証明書の責任者がいるか | 支払い忘れ、期限切れ、退職者依存でサービスが停止する | 個人依存を避け、更新通知、複数管理者、引き継ぎ手順を用意する |
| 権限、秘密、依存関係、復元を定期的に見直すか | 公開時は安全でも、時間と変更により設定が劣化する | 定期レビューと期限付きタスクを設定する |

# 10. 機能品質・データ品質

| チェック項目 | ダメな理由 | 対処方法 |
| --- | --- | --- |
| 主要シナリオを最初から最後まで試したか | 部分ごとのテストが通っても、利用者の一連の操作で失敗することがある | 作成者以外が登録、利用、変更、退会などを通しで操作する |
| 異常値と境界値を試したか | 未入力、長文、0件、大量件数、重複、通信切断で壊れる | 入力パターンと期待結果を整理し、自動・手動テストを追加する |
| 読込中、0件、権限なし、404、500の表示があるか | 画面が固まったように見え、利用者が次の行動を判断できない | 状態ごとのメッセージ、再試行、戻る導線、問い合わせ先を用意する |
| 連打、戻る、再読込で二重処理されないか | 二重登録、データ消失、在庫ずれ、二重請求が起きる | ボタン制御だけに頼らず、サーバーで冪等性と一意制約を持たせる |
| 日付、時刻、タイムゾーン、通貨、端数が正しいか | 締切、予約、請求、集計が地域や時刻境界で誤る | 保存形式と表示タイムゾーンを決め、境界時刻と端数をテストする |
| 削除・上書き前に対象と影響が分かるか | 誤操作で重要データを失い、復元できない | 確認、取消、履歴、論理削除、バックアップを重要度に応じて使う |
| 本番データや本番決済を使わずテストできるか | テストが実利用者、売上、通知へ影響する | 検証環境、テストアカウント、Sandbox、ダミーデータを用意する |
| 認証、メール、Webhook、ファイルを実際の接続で試したか | モックだけでは、権限、DNS、Secret、外部設定のミスに気づけない | 本番に近い検証環境でエンドツーエンドに確認する |
| 集計、料金、在庫などを元データと照合したか | コードが動いても、計算式や条件の誤りで業務上の損失が出る | 代表ケースを手計算・別集計と比較し、回帰テストを作る |
| 問い合わせ・障害・メンテナンスの導線があるか | 利用者が困っても連絡できず、運用者も停止や告知ができない | 問い合わせ先、ステータス表示、メンテナンスモードを用意する |

# 11. レスポンシブ・ブラウザ・アクセシビリティ

| チェック項目 | ダメな理由 | 対処方法 |
| --- | --- | --- |
| スマホ、タブレット、PCで主要画面を確認したか | PCだけで作ると、スマホで横スクロール、重なり、操作不能が起きる | 320px前後から大画面まで、実機または端末表示で確認する |
| 縦横、長文、文字拡大でも崩れないか | 実データやアクセシビリティ設定で、見切れや重なりが起きる | 折返し、可変高さ、200%ズーム、長い日本語・英数字を試す |
| Chrome以外でも主要機能が動くか | SafariやFirefox固有の差で、表示・入力・認証が失敗する | Safari、Firefox、iOS、Androidで重要シナリオを確認する |
| キーボードだけで操作できるか | マウスを使えない利用者が機能へ到達できない | 正しいHTML要素、フォーカス表示、順序、モーダルのフォーカス管理を行う |
| 画像とアイコンに読み上げ可能な説明があるか | スクリーンリーダー利用者に内容やボタンの目的が伝わらない | 情報画像に`alt`、装飾画像に空alt、アイコンボタンに名前を付ける |
| フォームにラベルと具体的なエラーがあるか | 入力内容や修正箇所が分からず、送信できない | `label`、説明、エラーの関連付けを行い、色だけで伝えない |
| 文字と背景のコントラストが十分か | 弱視、屋外、低品質画面で内容を読めない | WCAGを目安に色を測定し、文字拡大でも情報を失わないようにする |
| 見出し構造と言語指定が正しいか | 読み上げや検索エンジンが文書構造と言語を理解できない | 見出し順、ランドマーク、意味あるリンク文、`lang="ja"`を設定する |
| タップ対象が小さすぎないか | スマホで誤タップし、重要操作を間違える | 十分な大きさと間隔を確保し、隣接する操作を離す |
| 自動再生や大きな動きを制御できるか | めまい、注意散漫、操作妨害につながる | 停止手段を設け、`prefers-reduced-motion`へ対応する |

# 12. 画像・性能

| チェック項目 | ダメな理由 | 対処方法 |
| --- | --- | --- |
| 画像をBase64のdata URLで大量にHTML/CSSへ埋め込んでいないか | HTMLが肥大化し、画像だけをキャッシュできず、通信と解析が遅くなる | 画像ファイルをオブジェクトストレージやCDNへ置き、通常のURLで配信する |
| 画面サイズに合う画像を配信しているか | 小さな表示に巨大画像を読み込み、通信量と表示時間が増える | WebP/AVIF、複数サイズ、`srcset`、画像最適化サービスを使う |
| 画像の表示サイズを事前に確保しているか | 読み込み後に画面がずれ、誤タップやCore Web Vitals悪化が起きる | width/heightまたはaspect-ratioを指定する |
| 遅延読み込みを適切に使っているか | 全画像の即時読込は遅く、重要画像の遅延は最初の表示を悪化させる | 画面外画像はlazy、ファーストビューの重要画像は優先読込にする |
| 不要なJavaScript、CSS、フォントを含めていないか | 初期表示、操作可能までの時間、端末負荷が悪化する | バンドルを分析し、未使用コードと外部タグを削除・分割する |
| キャッシュと圧縮を使い分けているか | 静的資産は遅くなり、個人データは漏洩する危険がある | 静的資産はCDN・長期キャッシュ・圧縮、個人データはprivate/no-storeにする |
| 一覧で全件取得していないか | データ増加後に画面、API、DBが急激に遅くなる | ページング、件数上限、DBインデックス、必要列だけの取得を使う |
| N+1クエリや遅いSQLがないか | 件数に比例してDBアクセスが増え、本番データ量で停止する | クエリ計測、JOIN/一括取得、適切なインデックスを使う |
| 実回線でCore Web Vitalsを測ったか | 高速な開発PCでは、利用者の遅さや操作遅延に気づけない | スマホ回線、Lighthouse、実ユーザーデータでLCP/INP/CLSを確認する |
| 遅い箇所を特定できるか | 障害時にフロント、API、DB、外部サービスのどこが原因か分からない | トレース、APM、クエリログ、Web Vitalsを関連付ける |

# 13. SEO・公開情報・メール

| チェック項目 | ダメな理由 | 対処方法 |
| --- | --- | --- |
| 公開ページに固有のtitle、見出し、説明があるか | 検索結果や共有時に内容が分からず、重複ページとして扱われる | ページ内容に合うtitle、h1、必要なmeta descriptionを設定する |
| canonicalとnoindexを使い分けているか | 重複URL、検索結果、管理画面、エラーが検索へ登録される | 正規URLを指定し、非公開・低価値ページは認証またはnoindexにする |
| 本番サイト全体にnoindexが残っていないか | 公開しても検索結果へ表示されない | 本番設定、レスポンスヘッダー、robots metaを確認する |
| robots、サイトマップ、HTTPステータスが正しいか | クロールできない、削除ページが残る、エラーが正常ページ扱いになる | `robots.txt`、XMLサイトマップ、404/500のステータスを確認する |
| OGP、favicon、共有表示を確認したか | SNSやチャットで誤った画像・タイトル・URLが表示される | 主要ページのOGP画像、説明、canonical URL、faviconを実際に確認する |
| 構造化データが画面内容と一致しているか | 誤った商品、価格、評価を検索エンジンへ伝え、ペナルティや誤表示になる | Googleの仕様と検証ツールを使い、見えている事実だけ記述する |
| 会社情報、料金、解約、返金、必要な法定表示があるか | 利用者トラブル、返金紛争、法令違反につながる | サービス内容に応じて法務・担当者へ確認し、分かりやすく公開する |
| プライバシーポリシーが実態と一致しているか | 取得・AI利用・外部送信を説明せず、信頼低下や法的問題になる | 取得情報、目的、送信先、保存期間、削除、問い合わせ先を記載する |
| SPF、DKIM、DMARCを設定しているか | メールが迷惑メールになり、ドメインを詐称されやすい | DNSとメール配信サービスを設定し、実際の受信結果を確認する |
| ユーザー入力から任意のメールを送れないか | スパム踏み台、ヘッダ注入、大量送信に悪用される | 宛先・件名・テンプレート・回数をサーバー側で固定・制限する |

# 14. 決済を扱う場合

| チェック項目 | ダメな理由 | 対処方法 |
| --- | --- | --- |
| カード番号を自分のサーバーで保存・処理していないか | 情報漏洩の被害とPCI DSSの負担が大きくなる | Stripe Checkout/Elements等のホスト型・トークン化方式を使う |
| テストキーと本番キーを分けているか | テスト操作が実決済になり、本番キー漏洩の危険も増える | 環境ごとにキー、Webhook、商品、アカウントを分離する |
| 金額をブラウザからの値だけで決めていないか | リクエストを改ざんして、安い金額や不正な割引で購入される | サーバー側の商品マスタ、価格、通貨、割引条件から計算する |
| 完了画面だけで支払い済みにしていないか | URLを直接開く、画面改ざん、通信失敗で未払いを支払い済みにできる | サーバーで決済事業者のPayment状態を確認する |
| 決済Webhookの署名を検証しているか | 偽の決済成功通知を送られ、商品や権限を不正取得される | 公式ライブラリでraw bodyと署名を検証する |
| 二重決済を防いでいるか | 連打、通信再送、Webhook再送で同じ請求が複数回発生する | idempotency key、注文IDの一意制約、イベント重複排除を使う |
| 決済とDBの不整合を検知できるか | 支払い成功・注文失敗、またはその逆が起き、売上や提供状態がずれる | 状態遷移、再処理、定期照合、アラートを用意する |
| 失敗、3D Secure、返金、解約をテストしたか | 正常決済だけでは、実運用の例外で利用者と会計が不整合になる | Sandboxで失敗カード、認証、期限切れ、返金、サブスク更新を試す |
| 料金、解約、返金、税、請求書を明示しているか | 利用者トラブル、会計不整合、法令違反につながる | 担当者と条件を決め、購入前・管理画面・利用規約へ表示する |
| PCI DSSの適用範囲を確認したか | 決済代行を使っていても、事業者側に必要な手続きが残る場合がある | 決済事業者の案内に従い、必要な自己点検と運用を行う |
| 返金や権限変更など管理操作を保護しているか | 管理者乗っ取りや誤操作で金銭被害が発生する | MFA、再認証、最小権限、承認、操作ログを使う |
