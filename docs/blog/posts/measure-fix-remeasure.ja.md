# モデルが繰り返し書く「間違い」は、言語の欠陥の測定値である

2026-07-13 · Ailex 開発ノート #1 · [English](post.html?p=measure-fix-remeasure)

Ailex は「AI が書くこと」を前提に設計した小さな型付き言語だ。作りながら、AI に実際に書かせて測る、を繰り返している。この記事は、その測定で二度も同じ形で現れたパターンについて書く。結論を先に言うと——**AI が繰り返し書く不正なコードは、モデルの誤りではなく、言語の欠落を教えてくれる測定値だった。**

## 一度目：ラムダの型注釈

初期の Ailex では、無名関数の引数に型注釈が必須だった。`fn (acc : Float, x : Float) => acc + x` のように。設計としては真面目な選択のつもりだった。型は明示するほど安全、と。

早見表（言語仕様1ページ）だけを渡して、Claude Haiku と Claude Opus に8つの関数を書かせた。結果は Haiku が 8問中4問、Opus が 8問中5問の一発正解。そして**失敗の全件が、たった一つの同じ原因**だった：両モデルとも、JavaScript や Python の習慣のまま `fn (acc, x) => ...` と**注釈なしで**書いたのだ。

意味はほぼ常に合っていた。構文エラーの指摘を返せば1回で直る。つまりモデルは課題を解けている。解けていないのは、私の言語の構文だった。

言語側を直した。期待型が文脈から分かる位置では注釈を省略できるようにした（型推論）。再測定の結果：**両モデルとも 8/8、一発正解、修復ゼロ。** しかもコストが3割下がった。修復のやり取りが消えたからだ。

## 二度目：Option と map

言語が育って、`Option[T]`（値の不在を型で表す仕組み）を入れた後のこと。受け入れテストを16問に拡張して再測定したら、また同じ形が出た。

Haiku 15/16、Opus 15/16。そして失敗したのは**両モデルとも同じ1問、同じ書き方**だった：

```
unwrapOr(map(find(es, fn (e) => e.amount > lim), fn (e) => e.cat), "none")
```

`find` の結果（Option）に `map` を適用している。Rust の `option.map` や Haskell の `fmap` の感覚だ。当時の Ailex では `map` はリスト専用で、これは型エラーになる。「正しい」書き方は、番兵のダミー値を使って Option の中身を覗く不格好なものだった。

面白いのはここからで、**3ラウンドの型エラーのフィードバックを与えても、どちらのモデルも「正しい」書き方に辿り着かなかった。** 不自然な形を書くことを、事実上拒否した。

`map` を Option にも効くようにした（1関数の多重定義）。再測定：**両モデル 16/16。** nameFirstBig で両モデルが書いた式は、修正前と**一字一句同じ**——今度はそれが正解になった。

## 教訓の一般形

| | 修正前 pass@1 | 言語を修正 | 修正後 pass@1 |
|---|---|---|---|
| ラムダ注釈 | 50–63% | 注釈を任意化（推論） | **100%** |
| map × Option | 94% | map を Option に多重定義 | **100%** |

二度とも、直したのはモデルではなくプロンプトでもなく、**言語**だった。そして二度とも、修正後はモデルが最初から書いていた自然な形がそのまま正解になった。

人間向けの言語設計では「書き手を教育する」という選択肢がある。スタイルガイド、リンタ、コードレビュー。だが書き手が LLM のとき、その事前分布（何十億行の既存コードで形成された「自然な書き方」）は、こちらから変えられない定数だ。だったら、**測って、言語を合わせるほうが速い**。

> AI 向け言語の構文は、モデルの事前分布と喧嘩しないことが第一級の設計制約である。

## 正直な限界

- 測定は各16タスク・反復1回。規模は小さい。
- 修正と同時に早見表も1行変わるため、言語効果と教材効果は完全には分離できていない（ただし失敗が全件単一原因だったことから、主因は言語側と考えている）。
- 「モデルに合わせる」が常に正しいわけではない。安全性を落とす方向の prior（例：暗黙の型変換）には合わせない。Int と Float の混在は今も明示変換が必要だ。
