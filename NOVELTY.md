# Ailex 新規性の再監査（実測後・2026-07-10）

> 前提: 先行研究監査は [PRIOR_ART.md](PRIOR_ART.md)、実測は [EXPERIMENTS.md](EXPERIMENTS.md)。
> 本書は「実測(E1–E3, Q1, Q1b)を経て、当初の4中核アイデアの新規性がどうなったか」を再評価する。
> 新しい文献収集はしていない（既取得の監査＋実測の突き合わせ）。

## 要旨

当初の4中核アイデアは、**先行研究に取られているか、実測で価値が否定された**。
ただし実測の過程で、より新しく防御可能な「芯」が浮かんだ：**「言語設計の介入がいつ効き、いつ無意味になるか」を分離する対照研究**。

## 各中核アイデアの再判定

| 中核アイデア | 当初の主張 | 最近接先行 | 実測の証拠 | 判定 |
|---|---|---|---|---|
| **#1 単調型付け可能性** | 型エラーが構造的に出力不能 | ETH "Type-Constrained Code Gen" PLDI'25（機構を達成） | E1–E3: マスク生成は動くが Python+修復に勝てず（天井）、入力トークン 7–9x | 機構は先行に取られ、実測で優位を示せず |

## 実測が示したメタ発見（新しい芯の種）

3つの実験群を横断すると、一貫したパターンが見える：

1. **モデルの能力内のタスクでは、どの言語設計介入も効かない（天井効果）。** E1（Opus 10/10 vs Python 10/10）、compare 全般。
2. **介入が効くのは「コールドスタート／未知 API」regime だけ。** Q1: Haiku が未知の `dot` を当てられないときだけ差が出た（2/10 タスク）。
3. **そこで効くのは「何が使えるかの情報」であって、構造的制約でも機械可読形式でもない。** Q1b（内容 Δ+10、形式 Δ+0）。

**一言でいうと:**
> AIネイティブな言語設計の介入（制約付きデコード／構造化フィードバック／typed hole）は、
> **モデルが強くなる／ドメインが既知になるほど効果が消える。**
> 残る価値は未知ドメインでの「情報提供」であり、構造的制約それ自体ではない。

多くの論文は「うちの介入は効く」と主張する。**「いつ効き、いつ無意味になるか」を対照実験で分離した研究は薄い**（否定的・条件付き結果は出版されにくい）。ここが空き地。

## 新規性の芯の候補（再配置・優先度順）

### A.【最有力・即実行可】介入 × モデル能力 × ドメイン既知性 の対照研究
- 問い: constrained decoding / スコープ提示 / typed-hole の効果は、モデル能力（Haiku → Sonnet → Opus → Fable）とドメイン既知性（既知 API vs 未知 API）を振ると、**どこで消えるか**。
- 我々の harness（compare.ts / q1.ts）で**今すぐ測れる**。TEACH と MODEL を振るだけ。
- 新規性 = 個別介入の主張でなく「**介入が効く条件の地図**」＋それを測る**対照 testbed**。否定的結果を含めて誠実。
- Ailex を「より良い言語」でなく「**言語設計が LLM を助ける条件を測る testbed**」に再定義する。当初ビジョン（指示理解・推論・実行の精度を高める）を「どの層のどの介入が、いつ精度を上げるか」という測定問題に翻訳できる。

### B. by-design monotone typability vs ETH の retrofit
- 「言語設計として型安全生成を内在させると、既存言語への後付け(ETH)に対し実利（精度/コスト/実装単純さ）があるか」。
- ETH に機構は取られているので、示すべきは差分の実利。芯としては細い。

### C. 指示理解層: 軽量契約IR vs 自然言語（当初ビジョンの本丸）
- Vericoding は「重い形式仕様では NL は効かない」を示した。型＋契約＋実例の**軽量**中間が NL よりコード生成を上げるか、は中間帯。
- ただし spec-driven codegen は crowded。新規性は要・再監査（本書ではその判断材料を持たない）。

### D. 推論層: 計画の単調性（行動列の任意接頭辞が妥当な部分計画）
- #1 のアイデアをコードでなく計画に適用。未検証の空き地の可能性はあるが、先行（PDDL+LLM 系）の精査が必要で、build も大きい。

---

## 推奨

**芯を A に置く。**

理由:
1. 実測データが既にその方向（介入は条件付きでしか効かない）を示している。
2. harness が既にあり、追加コストが小さい。
3. 「介入が効く条件」を分離する対照研究は先行が薄く、否定的結果を含めて誠実（＝あなたの流儀）。
4. Ailex を「testbed」に再定義でき、当初の3層ビジョン（指示理解・推論・実行）を「各層の各介入が、いつ精度を上げるか」という一貫した測定問題にまとめ直せる。

**正直な但し書き:** これは「新しい言語 Ailex」を主役から降ろし、「言語設計介入の効き方を測る研究」に軸足を移す判断である。Ailex は主役でなく実験装置になる。新規性はより堅い（否定・条件付き結果は取られていない）が、当初の「世界初の言語」的な華やかさは薄れる。どちらを取るかは価値観の選択。

## 追補: 指示層＋貫通の的絞り再監査（2026-07-10, 狙い撃ち検索）

deepリサーチ不使用・WebSearch 4回＋WebFetch 1回で確認。結論は**当初想定より厳しい**。

### 指示層（意図IR / 契約IR）は飽和領域

- **"Intent Formalization: A Grand Challenge for Reliable Coding"**（arXiv:2603.17150, 2026）が、意図の形式化を**公式に grand challenge として枠づけ**。「AI-generated code is plausible by construction but not correct by construction」。
- 既存: **TiCoder**（テストで意図を形式化、開発者の正答評価が約2倍・arXiv:2208.05950 / 2404.10100）、LLM生成の事後条件/不変条件、Dafny/F*/Verus/3DGen、Spec-Driven（AWS Kiro, Spec Kit）、**Self-Spec**（モデル自著仕様）。
- 同論文は意図を **spectrum**（テスト→コード契約→論理契約→DSL）として整理。**Ailex の「型＋契約＋実例の軽量IR」は、この spectrum の一点にすぎず、それ自体は新規でない**（＝この分野の前提）。
- ただし同論文が挙げる**7つの未解決**は本物の空き: ①自己完結関数を超えるスケール(副作用/状態/並行) ②**オラクル無しでの仕様検証**（最重要）③**変更意図/編集**（新規作成でなく既存改変）④仕様のコスト効率ランキング ⑤豊かな論理(量化子/再帰) ⑥HCI ⑦ワークフロー統合。

### 貫通（層またぎ早期検証）も既に占有 ← 憲章の芯の弱点

- **VERIMAP**（Megagon）: verification-aware planning。タスクをDAG分解し、**SPEC/PLAN/TASKS をコード生成の前に検証して、幻覚API・不正パス・構造不整合を早期に捕まえ、実装後の実行検査も残す**。＝**憲章で基盤の芯に据えた「エラーを最も早い層で捕まえる／層またぎ検証」を、ほぼそのまま実装している。**
- **VeriGuard**（Google）: 検証済みコード生成でエージェント安全性。**"Code as Agent Harness"**: コードを推論・行動・検証の統一基盤とする視点。**PAT-Agent / NL→Kripke・LTL**: 計画の形式化と model checking。**Spec Kit Agents**: SPEC/PLAN/TASKS の文脈接地ワークフロー。

### 正直な結論

- **指示層・推論層・貫通のいずれも、2024–2026 に活発かつ飽和。** 「層をまたぐ早期検証の中間表現」という基盤の芯は、VERIMAP 等に先を越されている。
- したがって**概念レベルの「世界初」は現実的に取れない**（この空間は速く埋まった）。
- **残る現実的な差別化は概念でなく“技術的スタンス”**: 既存は多くが「オーケストレーション枠＋生成した検証関数（それ自体オラクル無しで誤りうる）」。Ailex 固有にできるとすれば「**単一の型付き・内容アドレスIRに、形式的な型/検証規律を層またぎで持たせる**」＝枠組みでなく言語/基盤としての構造的保証。ただしこれは細く・未確認・build 重い。
- **開いている具体的サブ問題（grand challenge の未解決から）**: オラクル無し仕様検証／変更・編集意図／仕様ランキング。ここは概念でなく問題が空いている。

### 含意（芯の再々配置）

1. **研究の新規性が欲しいなら**、飽和した中核でなく、上記の**開いたサブ問題を1つ**狙う（例: 変更意図の中間表現、オラクル無し仕様の検証）。
2. **有用な基盤を作りたいなら**、新規性より**動く実装＋明確な技術スタンス**（型付き内容アドレスIR）で勝負する（＝infra/product 価値。LLVM 的立ち位置）。
3. どちらにせよ、憲章の「層またぎ早期検証」単独の新規性主張は下方修正が必要（VERIMAP と重複）。
